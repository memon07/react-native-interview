How JSI Replaces the Bridge¶

The explanation interviewers want

The old bridge was an asynchronous, batched, JSON-serialized message queue between the JS thread and the native thread. Every setState touching native UI required JSON.stringify → bridge queue → deserialize → ViewManager dispatch.

JSI is a lightweight C++ API (jsi::Runtime, jsi::Value, jsi::Object, jsi::Function, jsi::HostObject) on top of Hermes. Native code registers HostObjects and HostFunctions that JS sees as ordinary objects. No serialization, no queue, optional synchronous call.

  • Memory model: HostObjects are reference-counted in C++ but handles are owned by the Hermes runtime. Misuse (capturing a jsi::Object past the runtime's lifecycle) is the #1 source of JSI crashes — a known interview topic.
  • Threading: JSI is single-runtime-per-thread. The JS runtime runs on the JS thread; Reanimated's UI Runtime is a second Hermes runtime on the UI thread. Hermes uses the Hades mostly-concurrent generational GC (short STW pauses).

Fabric vs Paper¶

  • Paper: JS → bridge → UIManager (mutable native view tree) → Yoga layout → main-thread mount.
  • Fabric: JS → Codegen-generated C++ component descriptors → immutable shadow tree in C++ → Yoga 3.0 layout on a background thread → mount commands diffed and applied on the UI thread → integrates with Concurrent React (interruptible renders, double-buffered trees).

The shadow tree being immutable enables concurrent rendering and safer cross-thread reads.

Hermes vs JSC¶

  • Hermes: AOT bytecode via hermesc, mmap'd at load. Hades GC. No JIT. Optimized for cold-start and low memory. Mandatory under New Arch. Incompatible with RAM bundles.
  • JSC: Interpreter + tiered JIT — better steady-state, worse cold start, bigger memory. Deprecated by Expo Go in SDK 52.

Production Migration Data¶

  • Shopify (Thiago Magalhaes, Sept 5, 2025): ~10% Android cold-start improvement, ~3% iOS. Some complex components 20% slower (New Arch exposed design cracks). Crash-free sessions briefly dipped below 99.95% before recovering. 40+ modules were deliberately not migrated to TurboModules during rollout.
  • Shopify (Mustafa Ali, Nov 2024): Migration eliminated 1.8M lines of redundant code; 86% code unification across platforms.
  • Discord × Margelo (2025): Jank frame rate on Android reduced by 26%. RN 0.79: TTI reduced by 400 ms (12% speedup, Samsung A14, one-line change).
  • Meta (Oct 23, 2024): 850+ libraries compatible; production-deployed in Facebook and Quest apps. No specific per-app cold-start numbers published.
  • Microsoft (May 9, 2025): Office on RNW runs Fabric on Windows App SDK; RNW 0.82 removed Paper. No public quantitative numbers.
Common Migration Pitfalls
  1. Mixing legacy bridge modules with TurboModules without correct threading — can deadlock on iOS main queue.
  2. Hermes + RAM bundles — incompatible. Use Metro inlineRequires: true instead.
  3. Third-party libs not New-Arch compatible — react-native-maps required 1.21.0+ for SDK 53.
  4. Codegen drift — change a spec, forget to re-run and clean → missing symbols. Lint specs in CI.
  5. JS-number precision loss — always use strings for 64-bit IDs; JS doubles lose precision silently.

EAS Update (OTA)¶

  • Channels embedded at build time; branches dynamic; channel-to-branch via eas channel:edit.
  • runtimeVersion must match — any native change requires a new binary.
  • Rollout: eas update --branch production --rollout-percentage 10 → escalate; monitor Sentry/EAS dashboard.
  • Rollback: eas update:rollback re-publishes a prior update.
  • Caveats: native code never goes OTA; offline users keep cached version. CodePush deprecated end of 2025.

Expo Router v3+ vs React Navigation v7¶

  • Expo Router = file-based routing on top of React Navigation. Folder structure = nav structure. _layout.tsx, groups (auth), dynamic [id], API routes +api.ts. Auto-generated typed routes (ExpoRouter.Routes).
  • React Navigation v7 (Nov 2024) added static config API with auto-inferred types.
  • Senior recommendation: Expo Router for greenfield; React Navigation v7 for brownfield or highly bespoke transitions.

EAS Build vs Fastlane¶

EAS owns credentials, parallel matrix builds, integrated EAS Update/Submit. Fastlane wins on flexibility and self-hosted control. Most teams: EAS for Expo apps, Fastlane for bare RN, sometimes both.

4. Performance Optimization¶

Startup Time Techniques¶

  1. Enable Hermes (mandatory under New Arch).
  2. Enable inline requires in Metro: inlineRequires: true.
  3. Move heavy init off the critical path; InteractionManager.runAfterInteractions.
  4. Lazy-load screens via dynamic imports.
  5. Tree-shake — replace lodash with lodash/<fn>; moment (290 KB) → dayjs (8 KB).
  6. Single Android libreactnative.so (RN 0.76+).
  7. Never use RAM bundles with Hermes.

Reanimated 3/4 Internals¶

  • Worklets carry the 'worklet' directive; Babel plugin workletizes them onto a separate Hermes UI Runtime.
  • Shared values (useSharedValue) provide thread-safe shared memory between JS and UI runtimes.
  • runOnJS / runOnUI (older) and scheduleOnRN / scheduleOnUI (newer) marshal calls between runtimes.
Pitfall (Andrei Calazans, June 2024)

Reading SharedValue.value from the JS thread synchronously triggers executeOnUIRuntimeSync, blocking JS by 500 ms+ on initial render.

FlashList v2 vs FlatList¶

FlatList uses VirtualizedList — destroys and recreates components as they scroll, causing blank cells. FlashList (Shopify) uses view recycling — keeps ~30 component instances and updates data, mirroring RecyclerView/UICollectionView. FlashList v2 (2025) is a complete rewrite for New Arch: JS-only (no native dep), no estimatedItemSize required, uses Fabric's synchronous layout measurements. New-Arch-only.

React Native Skia (2025)¶

Architectural shift: SkiaDOM → Fabric reconciler (immutable display list). Per William Candillon's 2025 post on shopify.engineering: animation-time performance up to 50% faster on iOS and nearly 200% faster on Android; time to first animation frame nearly 200% better on both platforms; 13% code reduction while adding functionality.

5. System Design for Mobile (Senior)¶

Offline-First Architecture¶

Layered approach: TanStack Query (in-memory + persister) → MMKV (durable KV) → SQLite/WatermelonDB (relational, observable) → sync engine.

SolutionBest ForNotes
WatermelonDBWhen you control the backend (NestJS + Postgres via Supabase)Pull/push protocol you implement. Used in Share My Stack, Nozbe.
PowerSyncLowest engineering costManaged sync from Postgres/MongoDB/MySQL → SQLite via CDC.
ElectricSQL + TanStack DBOpen-source middle path—
CRDTs (Jazz.tools, Yjs)Conflict-free collaborative editing—

Conflict resolution strategies: last-write-wins, server-wins, three-way merge, CRDT.

Real-Time¶

  • WebSockets (Socket.IO, native WebSocket) for bidirectional. Heartbeats + exponential backoff. iOS background sockets get killed.
  • SSE for one-way server→client streams.
  • Long polling as graceful fallback.
  • iOS Background App Refresh ≠ Android WorkManager — constrained differently.

Push Notifications¶

  • APNs (Apple, p8 key auth) + FCM (Android, also APNs broker).
  • Server-side fanout: OneSignal, Notifee+custom, AWS SNS.
  • Deep linking: Universal Links (iOS) + App Links (Android); URL scheme as fallback. Expo Router v3 + expo-linking make this declarative.

OTA Update Pipeline (Whiteboard)¶

CI → bundle build → sign → upload to CDN → device check on launch → background download → apply on next start → telemetry. Critical: runtimeVersion policy (sdkVersion / appVersion / fingerprint), channel-to-binary mapping, signing keys, rollout %, success telemetry.

Mobile Security¶

  • Cert pinning: react-native-ssl-pinning — pin at least two SHA-256 hashes (primary + backup) for rotation.
  • Root/jailbreak detection: jail-monkey, Google Play Integrity API (replaces deprecated SafetyNet).
  • Secure storage: expo-secure-store (Keychain/Keystore), never AsyncStorage for secrets.
  • Code obfuscation: Hermes bytecode is a deterrent; add Jscrambler / ProGuard.
  • OWASP Mobile Top 10 — enumerate at least the top five.

Feature Flags¶

LaunchDarkly, Statsig, Split, ConfigCat, Firebase Remote Config, Unleash. Cache locally (MMKV) for offline boot; refresh on foreground. Distinguish kill-switches (boolean, instant) from experiments (bucketed, sticky per-user). Bind to Sentry/Amplitude.

6. Native Integration — Kotlin + Swift¶

TurboModule in Kotlin (Android)¶

  1. Define NativeMyModule.ts spec: export interface Spec extends TurboModule { method(arg: string): Promise<string> }
  2. Codegen (during gradle generateCodegenArtifactsFromSchema) emits NativeMyModuleSpec.kt.
  3. Implement: class MyModule(reactContext: ReactApplicationContext) : NativeMyModuleSpec(reactContext)
  4. Register via a TurboReactPackage (or BaseReactPackage).
  5. Run threading-sensitive work via Kotlin coroutines (CoroutineScope(Dispatchers.IO).launch { ... promise.resolve(result) }).

TurboModule in Swift (iOS)¶

Codegen doesn't yet support pure Swift directly — write a thin Obj-C++ shim (.mm) that implements getTurboModule: and forwards to a Swift class via the -Swift.h umbrella header.

  • MyUtil.h (Obj-C interface) → MyUtil.mm (Obj-C++ calling Swift) → MyUtilImpl.swift
  • Use GCD queues (DispatchQueue.global(qos: .userInitiated)); dispatch back to main before touching UIKit.
  • requiresMainQueueSetup returns NO to allow synchronous JSI methods.
  • Nitro Modules (Margelo / Marc Rousavy) — community alternative with first-class Swift/Kotlin, minimal boilerplate.

JSI HostObject Pattern¶

  • HostObjects implement get / set / getPropertyNames for dynamic-property JS surfaces.
  • Canonical example: react-native-mmkv (synchronous KV, replaces AsyncStorage).
  • Use when same logic runs on iOS + Android (MMKV pattern), or for stateful native resources.

Kotlin Coroutines + RN¶

Dispatchers.Main.immediate for UI; Dispatchers.IO for disk/network; Dispatchers.Default for CPU. Resolve/reject Promise on completion.

Swift async/await + RN¶

Wrap async functions in Task { ... } and bridge via Obj-C++: Task { do { let r = try await asyncWork(); resolve(r) } catch { reject(...) } }

7. TypeScript Advanced (Senior Level)¶

  • Generics with constraints: function pluck<T, K extends keyof T>(obj: T, key: K): T[K]
  • Conditional types + infer: type ReturnType<T> = T extends (...args: any) => infer R ? R : never
  • Mapped types: type ReadonlyDeep<T> = { readonly [K in keyof T]: T[K] extends object ? ReadonlyDeep<T[K]> : T[K] }
  • Discriminated unions for RN navigation: type RootStackParamList = { Home: undefined; Profile: { userId: string } }
  • Zod: schema → z.infer<typeof Schema> → Schema.safeParse(data) → handle discriminated union.
  • Type-safe API clients: tRPC (when you control backend), GraphQL Code Generator, OpenAPI Typescript generator.

FAANG TypeScript Traps¶

  • "Why does let x: string[] = [] infer never[] then break?" (Widening/narrowing.)
  • "What's the difference between any and unknown?" (unknown requires narrowing.)
  • "Why does as const make a tuple a tuple?" (Literal type widening prevention.)
  • Variance: function parameters bivariant by default; strictFunctionTypes makes them contravariant.
  • Declaration merging for module augmentation (Expo Router, MMKV).

8. CI/CD + DevOps for Mobile¶

GitHub Actions for RN¶

  • macOS runners are ~10× the cost of Linux — minimize iOS via path filters.
  • Cache ~/Library/Caches/CocoaPods, ~/.gradle/caches, node_modules.
  • Use gradle-build-action, setup-node with cache: yarn.
  • Sign with apple-actions/import-codesign-certs + apple-actions/download-provisioning-profiles.
  • Trigger EAS via expo/expo-github-action.

Key Findings¶

  1. The New Architecture debate is over. RN 0.76 made Fabric, TurboModules, Codegen and Bridgeless the default; RN 0.82 made the Legacy Architecture non-disableable. Expo SDK 55 cannot disable it. Senior candidates must explain JSI, shadow tree, threading, and migration tradeoffs from first principles.

  2. JSI is not just "a faster bridge." It is a C++ interface (jsi::Runtime) that lets native code expose HostObjects and synchronous functions directly to the Hermes JS VM. Serialization is gone for hot paths; Reanimated, MMKV, react-native-skia, and FlashList v2 are all JSI-native today.

  3. FAANG interview loops are uniform, with mobile as a substitution round. Meta E5 = 4 rounds; Amazon SDE-II = OA + 4×55-min onsite + Bar Raiser; Google L5 = 3 coding + 1 mandatory system design, with one DSA round optionally swappable for Android/iOS.

  4. Startup interviews invert the weighting: less LeetCode, more "design an offline-first feature that scales to 1M users on flaky 3G." Pragmatic shipping evidence carries more weight than DSA grind.

  5. Production migration data is mostly qualitative. Only Shopify (~10% Android cold-start improvement) and Discord/Margelo (26% jank reduction; 12% TTI speedup) have concrete published numbers. Don't invent numbers for Meta or Microsoft.

  6. The dominant senior toolchain in 2026: Expo SDK 53/54 + EAS Build/Submit/Update + Expo Router v3+ + Reanimated 3/4 + FlashList v2 + react-native-skia + MMKV + WatermelonDB or PowerSync + Sentry + LaunchDarkly/Statsig. Be opinionated about it.

Recommendations¶

Staged action plan with benchmarks that gate progression.

Stage 1 — Foundation (end of Week 4): Whiteboard JSI, Fabric, TurboModules, Codegen, threading, and Hermes for 20 minutes without notes. Have a Kotlin TurboModule and a Swift TurboModule running on the same TS spec.

Benchmark (answer all five without prep): (a) What does TurboModuleRegistry.getEnforcing do? (b) What thread does a Reanimated worklet run on, and where does its memory live? (c) Why are RAM bundles incompatible with Hermes? (d) How does Codegen find your TS spec? (e) What's the difference between Fabric's shadow tree and Paper's UIManager?
Stage 2 — System Design Depth (end of Week 8): Whiteboard "design Instagram offline" in 45 min, articulating sync strategy, conflict resolution, push notifications, observability, and security. One EAS-deployed app with OTA + Sentry shipped.

Benchmark: ≥6/8 on Hello Interview's mobile system design rubric in a mock. If not, two more mocks before progressing.
Stage 3 — Mock Loop Readiness (end of Week 11): Consistently passing peer DSA mocks at LeetCode Medium speed (≤25 min) and Hello Interview / Exponent system design mocks. STAR stories cover 7+ behavioral dimensions.

Benchmark to apply: ≥75% Medium LeetCode hit rate in 30 min cold; ≥80% in 45 min for one Hard.

  • Startup mocks reveal architecture gaps → flip the ratio.
  • Behavioral feedback says "too in the weeds" → record yourself; target 2-min STAR with explicit metrics.

Negotiation Prep (Week 12)¶

levels.fyi for FAANG comp; Patrick McKenzie's "Salary Negotiation." Know vesting schedules: Amazon 5/15/40/40 back-loaded — counter with sign-on; Meta even vest; Google with multipliers.

Caveats¶

  • The "Senior React Native interview at FAANG" framing is partially aspirational. No FAANG publishes a distinct Senior RN loop. Mobile is a specialization round. You will be tested on DSA and generic system design as much as RN internals.
  • Production performance numbers are scarce. Shopify and Discord (via Margelo) are the only companies with publicly available, named, dated, numeric New Architecture migration outcomes. The "43% faster cold start" figure from AgileSoftLabs (March 2026) cites unnamed migrations — treat as marketing copy.
  • The Kraken "2x speed, 5x render time" figure does not appear in Kraken's primary blog — only in third-party summaries. Don't cite it as primary.
  • Library compatibility is still uneven. Always run npx expo-doctor@latest before promising a migration timeline.
  • Microsoft CodePush has been deprecated; migrate to EAS Update.
  • AI-assisted coding rounds at Meta are new (2025). Practice with Cursor / Copilot under timed conditions. Not permitted at Google.
  • The 3-month plan is aggressive. With heavy day-job demands, extend to 4–5 months rather than skip system design or native module work.
  • Conference talks age quickly. Pre-mid-2024 content likely predates New Arch default. Prioritize App.js Conf 2024+, Chain React 2024+, React Conf 2024+.
  • Expo lock-in is real but reversible. Be ready for "what would you do if Expo's pricing changed dramatically?" — prebuild + bare workflow + Fastlane is the escape hatch.
