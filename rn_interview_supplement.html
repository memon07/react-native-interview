Arrays & Sliding Window

Medium Longest Substring Without Repeating Characters Meta · LC#3
Pattern: Sliding Window + Set Time: O(n) Target: ≤15 min

Why it's asked at Meta: Tests core sliding-window intuition. Appears frequently in Meta phone screens.

TypeScriptfunction lengthOfLongestSubstring(s: string): number {
  const seen = new Map<string, number>();
  let left = 0, max = 0;
  for (let right = 0; right < s.length; right++) {
    if (seen.has(s[right]) && seen.get(s[right])! >= left) {
      left = seen.get(s[right])! + 1; // shrink window
    }
    seen.set(s[right], right);
    max = Math.max(max, right - left + 1);
  }
  return max;
}

Key insight: store the index in the map, not just membership — lets you jump left pointer in O(1).

Medium Minimum Window Substring Amazon · LC#76
Pattern: Sliding Window + Frequency Map Time: O(n) Target: ≤25 min

Why it's asked at Amazon: Tests "have/need" counting pattern. Classic Amazon OA problem.

TypeScriptfunction minWindow(s: string, t: string): string {
  const need = new Map<string, number>();
  for (const c of t) need.set(c, (need.get(c) ?? 0) + 1);
  let have = 0, total = need.size;
  const window = new Map<string, number>();
  let [left, res, resLen] = [0, "", Infinity];
  for (let r = 0; r < s.length; r++) {
    window.set(s[r], (window.get(s[r]) ?? 0) + 1);
    if (need.has(s[r]) && window.get(s[r]) === need.get(s[r])) have++;
    while (have === total) {
      if (r - left + 1 < resLen) { res = s.slice(left, r + 1); resLen = r - left + 1; }
      window.set(s[left], window.get(s[left])! - 1);
      if (need.has(s[left]) && window.get(s[left])! < need.get(s[left])!) have--;
      left++;
    }
  }
  return res;
}

Trees & Graphs (High Frequency)

Medium Binary Tree Level Order Traversal All FAANG · LC#102
Pattern: BFS + Queue Time: O(n) Target: ≤12 min

Master template — dozens of variants (zigzag, right side view, max width) all follow this exact queue-drain pattern.

TypeScriptfunction levelOrder(root: TreeNode | null): number[][] {
  if (!root) return [];
  const result: number[][] = [], queue: TreeNode[] = [root];
  while (queue.length) {
    const level: number[] = [];
    let size = queue.length; // snapshot size BEFORE draining
    while (size--) {
      const node = queue.shift()!;
      level.push(node.val);
      if (node.left) queue.push(node.left);
      if (node.right) queue.push(node.right);
    }
    result.push(level);
  }
  return result;
}
Medium Number of Islands All FAANG · LC#200
Pattern: Graph DFS / BFS flood-fill Time: O(m×n) Target: ≤15 min

Master of the flood-fill pattern. Comes up in disguise: count connected components, find largest island, etc.

TypeScriptfunction numIslands(grid: string[][]): number {
  let count = 0;
  const dfs = (r: number, c: number) => {
    if (r < 0 || r >= grid.length || c < 0 || c >= grid[0].length
        || grid[r][c] !== '1') return;
    grid[r][c] = '#'; // mark visited in-place
    [[-1,0],[1,0],[0,-1],[0,1]].forEach(([dr,dc]) => dfs(r+dr, c+dc));
  };
  for (let r = 0; r < grid.length; r++)
    for (let c = 0; c < grid[0].length; c++)
      if (grid[r][c] === '1') { dfs(r, c); count++; }
  return count;
}
Hard Word Ladder Google · LC#127
Pattern: BFS shortest path on implicit graph Time: O(M²×N) Target: ≤35 min

Key insight: Model each word as a node; edges exist when words differ by one letter. Shortest path = BFS. The trick is generating neighbors via wildcard buckets, not comparing all pairs.

TypeScriptfunction ladderLength(begin: string, end: string, wordList: string[]): number {
  const wordSet = new Set(wordList);
  if (!wordSet.has(end)) return 0;
  const queue: [string, number][] = [[begin, 1]];
  const visited = new Set([begin]);
  while (queue.length) {
    const [word, dist] = queue.shift()!;
    for (let i = 0; i < word.length; i++) {
      for (let c = 97; c <= 122; c++) {
        const next = word.slice(0,i) + String.fromCharCode(c) + word.slice(i+1);
        if (next === end) return dist + 1;
        if (wordSet.has(next) && !visited.has(next)) {
          visited.add(next); queue.push([next, dist + 1]);
        }
      }
    }
  }
  return 0;
}

Dynamic Programming

Medium Coin Change Amazon · LC#322
Pattern: Bottom-Up DP (unbounded knapsack) Time: O(amount × coins) Target: ≤20 min 
TypeScriptfunction coinChange(coins: number[], amount: number): number {
  const dp = new Array(amount + 1).fill(Infinity);
  dp[0] = 0;
  for (const coin of coins)
    for (let a = coin; a <= amount; a++)
      dp[a] = Math.min(dp[a], dp[a - coin] + 1);
  return dp[amount] === Infinity ? -1 : dp[amount];
}

Fill dp[0]=0, everything else Infinity. For each coin, update forward. Classic "fewest coins" pattern.

Hard Longest Increasing Subsequence Google · LC#300 (O(n log n) variant)
Pattern: Patience Sorting / Binary Search Time: O(n log n) Target: ≤35 min

Interviewer signal: if you only know O(n²) DP, that's ok for Medium. Knowing the O(n log n) tails approach separates Staff candidates.

TypeScriptfunction lengthOfLIS(nums: number[]): number {
  const tails: number[] = [];
  for (const n of nums) {
    let lo = 0, hi = tails.length;
    while (lo < hi) {
      const mid = (lo + hi) >> 1;
      tails[mid] < n ? lo = mid + 1 : hi = mid;
    }
    tails[lo] = n;
  }
  return tails.length;
}

Worked Example 1: Design a News Feed (Instagram/Twitter)

Scope Assumptions (you negotiate these in the interview)

  • 50M DAU, read-heavy (100:1 read/write ratio), iOS + Android
  • Offline: show cached feed for up to 48h. New posts need connectivity.
  • Media: images only (no video for this scope)
  • Real-time: new post indicators, not live feed injection

API Contract

// Feed
GET  /feed?cursor=<opaque_token>&limit=20
     → { items: Post[], nextCursor: string | null, ttl: number }

// Post creation
POST /posts  { mediaUrls: string[], caption: string }
             → { postId: string, status: "processing" }

// Media upload (presigned S3)
POST /media/presign { mimeType: string }
     → { uploadUrl: string, cdnUrl: string }

Local Data Model (WatermelonDB)

// posts table
postId: string (indexed), authorId, createdAt (indexed),
caption, mediaUrls: JSON, localStatus: 'synced'|'pending'|'failed',
seenAt: number | null   // for "new posts" badge

// feed_cursor: MMKV
'feed:cursor'   → opaque server cursor string
'feed:lastSync' → epoch ms

Image Loading Pipeline

  1. Store blurhash string alongside CDN URL in the Post model
  2. Render <ExpoBlurHash> immediately from local DB — zero network latency
  3. expo-image fetches WebP from CloudFront with cachePolicy="disk"
  4. Prefetch next 5 images in FlashList's onViewableItemsChanged
  5. On low memory (AppState → background): call Image.clearMemoryCache()

Offline Strategy

  • Read path: Query WatermelonDB first. If lastSync < 48h, show cached feed with staleness banner.
  • Write path: Write to local DB with localStatus: 'pending'. Optimistically show in feed. Background sync queue retries on connectivity restore (NetInfo.addEventListener).
  • Conflict: Server is authoritative for published posts. Last-write-wins for draft edits.

Threading & Architecture

Q
Walk me through exactly what happens on the JS thread vs UI thread vs native modules thread in RN 0.76+. What changed with New Architecture?

Old arch (Bridge): 3 threads — JS thread (JS execution, React reconciliation), UI thread (native view updates), and Shadow Thread (Yoga layout). Communication was async JSON batches over the bridge. You could not call native modules synchronously.

New arch: JS thread still runs Hermes. The Shadow Thread is replaced by C++ Fabric renderer running concurrent layout on a background thread. TurboModules expose JSI HostObjects — native code can now be called synchronously from JS with zero serialization. Reanimated worklets run on a dedicated Worklet thread (a second Hermes runtime), sharing memory with the C++ layer via JSI — that's why animations survive JS thread jank. The UI thread still handles final view mutations, now via atomic Fabric commit.

Implication you should raise unprompted: With New Arch, blocking the JS thread can still cause missed frames because Fabric batches commits to the UI thread — but worklet-based animations are fully immune since they bypass JS. That's why Reanimated 3+ is preferred over Animated for any gesture-coupled animation.

Q
What is a JSI HostObject? Write a minimal C++ implementation.

A HostObject is a C++ class implementing jsi::HostObject that is exposed directly to the Hermes JS runtime. JS can call methods on it synchronously — no serialization, no async bridge, no JSON. MMKV, Reanimated, and react-native-skia are all HostObject-based.

C++class MyStorage : public jsi::HostObject {
public:
  jsi::Value get(jsi::Runtime& rt, const jsi::PropNameID& name) override {
    auto propName = name.utf8(rt);
    if (propName == "getString") {
      return jsi::Function::createFromHostFunction(rt, name, 1,
        [this](jsi::Runtime& rt, const jsi::Value&,
               const jsi::Value* args, size_t) -> jsi::Value {
          auto key = args[0].getString(rt).utf8(rt);
          return jsi::String::createFromUtf8(rt, store_[key]);
        });
    }
    return jsi::Value::undefined();
  }
private:
  std::unordered_map<std::string, std::string> store_;
};

Install into the runtime: rt.global().setProperty(rt, "MyStorage", jsi::Object::createFromHostObject(rt, std::make_shared<MyStorage>()));

Q
Describe the Fabric commit pipeline from setState to pixels on screen.

5-phase pipeline:

  1. Render phase (JS thread): React reconciler runs, produces updated Fiber tree. Host config calls emit mount instructions.
  2. Commit phase (JS thread → C++): Shadow tree is built/diffed in C++ (react/renderer/core). Yoga layout runs on a shadow tree clone. This can run concurrently with JS — that's Concurrent React in RN.
  3. Mount phase (UI thread): Fabric applies the committed shadow tree diff to native views atomically. View mutations are coalesced — no partial frame renders.

Key difference from Paper: In Paper, UIManager was JS-driven and views were updated via async bridge calls (could produce partial updates). In Fabric, the UI thread pull-syncs from the C++ tree atomically, eliminating the "tearing" artifact where layout and paint could be out of sync.

Memory & GC

Q
How does Hermes Hades GC work and why does it matter for animations?

Hades is a concurrent mark-sweep collector for the old generation. Unlike the previous stop-the-world GC, Hades runs the mark phase on a background thread concurrently with JS execution. The JS thread only stops briefly for final evacuation (typically <1ms). Old GC could pause the JS thread for 100–200ms — enough to drop multiple frames.

Why it matters for animations: Even with Reanimated worklets bypassing the JS thread, any GC pause on the JS thread affects state updates and re-renders driven from JS. With Hades, UI-coupled JS work (gesture state sync, scroll event handlers written in JS) is far less likely to jank. Hades became the default in Hermes for React Native in 0.74+. Up to 30× shorter pause times in benchmarks.

Mention in context: "If I see jank that doesn't correlate with JS thread work, I look at GC events in Hermes profiler output before assuming it's a rendering issue."

Q
What are SharedValues in Reanimated 3 and how do they cross the JS/worklet boundary without serialization?

A SharedValue (created with useSharedValue()) is a C++ object exposed to both the JS runtime and the Worklet runtime via JSI HostObjects. There is no serialization — both runtimes access the same underlying C++ memory via JSI pointers.

When you write sv.value = 1 from JS, it's a synchronous C++ write. When a worklet reads sv.value, it reads the same C++ memory synchronously. The Worklet runtime runs on its own thread (Reanimated's dedicated queue), so reads/writes from both sides are guarded by an internal mutex.

Implication: You can animate at 60/120fps driven entirely by native gesture events, with zero round-trips to the JS thread. The JS thread can be fully blocked and animations keep running. This is the fundamental reason Reanimated + RNGH replaces the old Animated.event pattern.

Architecture & Production

Q
How would you implement a safe, gradual rollout of a TurboModule migration across 5M users?

Never do a big-bang migration. Strategy:

  1. Parallel registration: Register both the legacy NativeModule and the TurboModule. Use a feature flag (LaunchDarkly/Statsig) to select which one is instantiated at runtime — TurboModuleRegistry.getEnforcing in the new path, NativeModules.MyModule in the legacy path.
  2. Canary (1%): Enable TurboModule for 1% of users. Watch Sentry for new crashes specifically correlated with that cohort. Monitor for 24–48h on a stable release.
  3. Ramp (10% → 50% → 100%): Each stage requires passing a crash-free session threshold (99.5% minimum). Build a kill-switch: feature flag can be turned off remotely without an OTA.
  4. Remove legacy path: Only after 100% ramp has been stable for 2+ releases.

Common trap: TurboModules change the threading model. Async callbacks that were "fire and forget" on the bridge now need explicit threading guarantees. Test on the lowest-spec device you support.

Q
What is Codegen and what does it actually generate?

Codegen is a build-time tool that takes your TypeScript spec file (a .ts file exporting TurboModuleRegistry.getEnforcing or codegenNativeComponent) and generates:

  • Android: Java/Kotlin abstract classes with type-safe method signatures that your native module extends
  • iOS: Objective-C protocol header (RCTMyModuleSpec.h) that your Swift/ObjC class conforms to
  • C++: JSI binding glue that connects the TS spec to the native implementations

The key benefit: type safety enforced at build time rather than runtime. If you change the JS spec without updating native, the build fails. Before Codegen, type mismatches produced silent runtime crashes. npx react-native codegen runs it; in Expo it's run during expo prebuild. The output lives in android/app/build/generated/source/codegen/ and ios/build/generated/ios/.

Q
Your app has a memory leak that only appears after 20+ minutes of use. Walk me through how you'd diagnose it.

Step 1 — Confirm it's a leak, not expected growth: Android Studio Memory Profiler / Xcode Instruments Leaks. Take heap snapshots every 5 min. If retained heap grows linearly without GC recovery → leak.

Step 2 — Isolate to JS vs native: Hermes profiler (Chrome DevTools → remote JS debugging) shows JS heap. If JS heap is stable but native heap grows → C++/native module leak. If JS heap grows → JS-side leak.

Common JS leaks: useEffect with missing cleanup (event listeners, subscriptions, timers), closures capturing large objects in module scope, cached data in custom stores never evicted.

Common native leaks: TurboModule holding a strong reference to an Activity/Context (use WeakReference on Android), iOS delegate cycles (use weak self in Obj-C callbacks).

Step 3 — React-specific: Use why-did-you-render + component profiler to find components that never unmount. Common in modal/overlay patterns where unmount is never called due to wrong navigation stack handling.

Use inline requires (Metro config) for the same effect with Hermes. Staff
What thread does a Reanimated worklet run on? Reanimated's dedicated Worklet thread — a separate Hermes runtime instance. It shares memory with the C++ layer via JSI SharedValues, but has its own call stack. It never touches the JS thread during animation execution. Senior
What's the difference between Fabric's shadow tree and Paper's UIManager? Paper's UIManager was a JS-driven, async queue of view mutations — views could be partially updated mid-frame. Fabric's shadow tree is a C++ immutable tree; diffs are committed to the UI thread atomically in one operation, preventing partial frame renders and tearing. Staff
How does TurboModuleRegistry.getEnforcing differ from NativeModules? NativeModules lazily initialized via the bridge at runtime with no type guarantees. TurboModuleRegistry.getEnforcing resolves synchronously via JSI at first access, throws immediately if the module is missing (hence "Enforcing"), and is type-safe via Codegen. Senior
When would you choose CRDT over last-write-wins for conflict resolution? When concurrent edits to the same field must merge without data loss — e.g., collaborative text editing, shared lists, counter increments. Last-write-wins is simpler but silently drops concurrent writes. CRDT (Yjs, Automerge) guarantees convergence. Use last-write-wins when one user owns the data (notes, profile fields). Staff
What is Play Integrity and when do you need it? Google's replacement for SafetyNet — attests that the APK is unmodified, running on a real uncertified Android device, and the app was installed from Play. Required for payment flows, high-value loyalty features, anything that can be gamed by rooted/emulated devices. Check the verdict server-side, never on-device. Senior
Explain Yoga layout and how Fabric uses it. Yoga is a cross-platform C++ layout engine implementing Flexbox. Fabric builds a shadow tree of Yoga nodes from the React element tree, runs layout (calculating positions/sizes) entirely in C++ on a background thread, then commits the final positions to native views on the UI thread. This removes the JS-thread layout bottleneck of Paper. Principal
What's a Baseline Profile on Android and what does it buy you? A static file bundled with the APK listing hot methods for ART's Ahead-of-Time compilation. Without it, ART interprets code until JIT kicks in after first runs. With it, startup-critical paths are AOT-compiled from first launch. Shopify reported ~10% Android cold-start improvement; RN community benchmarks suggest up to 40% in some configurations. Staff

⚠️

Always take at least 48 hours.  ·  Competing offers are your only real leverage — run parallel processes.

Phase 1
Research

First Offer

Counter

Brownfield Migration Depth

Large companies almost always have brownfield apps. Your docs mentioned section 9 briefly. Here: incremental migration strategies, SDK versioning, feature flag rollout patterns — the actual operational detail interviewers probe at Staff level.

The Incremental Brownfield Insertion Pattern

The correct brownfield strategy is never to rewrite. You insert RN as a navigation entry point into an existing native app, route screen by screen, and let the native shell remain the host.

Phase 1
Host Setup
  • Add react-native as a library dependency (not a standalone app) to the existing Xcode/Gradle project
  • Create a single RCTRootView (iOS) / ReactRootView (Android) host container — a native ViewController/Fragment that boots the RN runtime
  • RN runtime initializes once at app start (background thread) and is reused across all RN surfaces — critical for keeping cold start impact low
  • Navigation: native app's router remains the authority. RN screens are pushed as native ViewControllers with RN content inside
Phase 2
First Surface
  • Pick a low-risk, high-velocity screen: settings, onboarding, notifications center — not checkout, not auth
  • Define the native-to-RN interface early: how will native pass params into the RN bundle? (initialProperties on RCTRootView)
  • Define the RN-to-native callback interface: how will RN trigger native navigation events? (TurboModule with navigateToNativeScreen method)
  • Shadow DOM/layout budget: ensure RN screen's Yoga layout matches the native design system's spacing tokens
Phase 3
SDK Versioning
  • The problem: as you add RN surfaces, multiple product teams may need different native module APIs. You need a versioning contract.
  • Treat your TurboModules as internal SDKs: semantic versioning, changelog, deprecation window (minimum 2 releases)
  • Use a module registry pattern: one PlatformSDK TurboModule that exposes a version string; consumers gate behavior on version
  • Never remove a method from a TurboModule without a 2-release deprecation period — RN OTA updates can run ahead of native releases
Phase 4
Feature Flags
  • Every RN surface ships behind a feature flag from day one — allows instant kill-switch without OTA or app release
  • Flag is evaluated native-side before the RN surface is even instantiated — eliminates the "blank screen flash" from flag evaluation latency in JS
  • Tiered rollout: internal → 1% → 10% → 50% → 100% with automated crash-rate gates between each tier
  • Post-100% stable (≥2 releases): remove the flag and the native fallback to prevent flag debt

Multi-Version RN in a Monorepo

At large orgs, different product teams may be on different RN minor versions. This is a maintenance nightmare but a reality at companies like Microsoft.

🚨
Strongly Avoid Multiple RN Versions in One Binary
Two RN runtimes in one APK/IPA causes duplicate Hermes instances, doubled cold start overhead, and impossible native module version conflicts. The correct approach is a single RN version, centrally upgraded on a cadence. Use a Mobile Platform team to own the upgrade; individual product teams are consumers, not version owners.

The "Upgrade Cadence" Pattern (Shopify / Microsoft approach)

  1. Version lock: All product teams use the exact same react-native version. It lives in the root package.json, not per-team.
  2. Upgrade train: Mobile Platform team upgrades RN every 1–2 minor versions. They own the compatibility shim layer for any breaking native module changes.
  3. Migration window: 6 weeks for product teams to update any breaking API usages. Breaking changes are announced in an internal RFC with migration guide.
  4. Codemods: Where possible, the Platform team ships automated codemods (jscodeshift) for JS-side breaking changes.

Native Navigation Protocol (for interviewer depth)

One of the most underspecified areas in brownfield RN. When an RN screen needs to navigate to a native screen, you need a clean contract:

TypeScript// TurboModule spec: NativeNavigator.ts
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  navigate(screenName: string, params: Object): void;
  dismissRN(): void;       // close RN modal/sheet
  replaceWithNative(screenName: string, params: Object): void;
}

export default TurboModuleRegistry.getEnforcing<Spec>('NativeNavigator');

If you can speak to all five, you've demonstrated organizational scope that most senior candidates haven't thought about.
