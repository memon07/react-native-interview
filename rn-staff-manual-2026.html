Activity Lifecycle — The Complete Picture¶

Most engineers know onCreate/onResume. Staff engineers know why they exist and how process death, configuration changes, and back-stack interact.

  App Launch
      │
      ▼
  onCreate()  ←──────────────────────────────┐
      │         (savedInstanceState)          │
      ▼                                       │
  onStart()                                   │ Activity
      │                                       │ Recreated
      ▼                                       │ (rotation,
  onResume()  ◄── VISIBLE & INTERACTIVE       │  locale change)
      │                                       │
      │ (another activity comes to foreground)│
      ▼                                       │

  onStop()  ─── release heavy resources       │

      │

Critical
Process Death — The Silent Killer
Android can kill your process while it's in the background. When the user returns, the OS recreates the Activity from savedInstanceState. Any in-memory state (ViewModel data, global singletons) is gone. React Native apps frequently crash on restoration because JS state isn't persisted. Staff engineers ensure persistence through SharedPreferences, Room, or MMKV + proper ViewModel + SavedStateHandle.

Android Services¶

Foreground Service
Runs with persistent notification. Required for audio playback, location tracking, file downloads. RN: react-native-track-player uses this. Android 14+ requires declaring foreground service type in manifest.
Background Service
Deprecated for long work. Use WorkManager instead. Background services are killed aggressively on API 26+. Common mistake: using old IntentService for background sync in RN apps.
Bound Service
Allows IPC via AIDL or Messenger. Used for in-app billing, DRM, Bluetooth. RN TurboModules don't use this—they use JSI. Staff engineers understand AIDL for complex native integrations.

Looper & Handler Internals¶

Every Android thread can have a Looper — an infinite loop that dequeues Messages and dispatches them to a Handler. The main thread's looper is created by ActivityThread.main(). This is why UI must happen on the main thread.

Binder IPC — The Android IPC Backbone¶

Binder is Android's kernel-level IPC mechanism. Every system service call (startActivity, getSystemService) goes through Binder. Understanding it explains why main thread work causes ANRs — binder transactions timeout after 5 seconds.

ANR Internals & Debugging¶

Production IncidentANR Spike: 2% of users on cold start

Symptom: ANR in startup. /data/anr/traces.txt showed main thread blocked in SharedPreferences.getAll().

Root cause: RN app loaded large MMKV/SharedPreferences synchronously during ReactNativeHost.getPackages() initialization, which runs on the main thread.

Fix: Deferred initialization using lazy + moved to WorkManager pre-fetch. Used DataStore (async) instead of SharedPreferences for large data.

Tools used: Android Vitals, Perfetto, adb bugreport, simpleperf

R8 Gotcha in Production
R8 with full mode aggressively removes classes it thinks are unused. In RN, TurboModule codegen classes and JSI C++ native code are loaded via reflection/JNI — R8 doesn't see these references. Always verify your keep rules with ./gradlew :app:generateReleaseR8Usage and test release builds on real devices before shipping.

Multidex & Dexing¶

Android's Dalvik/ART limits a single DEX file to 65,536 method references (the "64K limit"). Large RN apps with many node_modules routinely exceed this. multiDexEnabled = true splits into multiple DEX files. On API 21+ this is handled natively. Below API 21 requires the multidex support library.

Coroutines — Deep Dive¶

Coroutines are not threads. They are suspendable computations — code that can pause without blocking a thread, resume later on any thread, and handle cancellation cooperatively.

Text
// Structured Concurrency: coroutines have parent-child relationships
class MyTurboModule(reactContext: ReactApplicationContext) :
    NativeMyModuleSpec(reactContext) {

    // Module scope: cancelled when module is torn down
    private val scope = CoroutineScope(SupervisorJob() + Dispatchers.IO)

    override fun fetchData(promise: Promise) {
        scope.launch {
            try {
                val result = withContext(Dispatchers.IO) {
                    networkClient.get("/data")  // suspending
                }
                // Switch to JS thread to resolve
                promise.resolve(result.toWritableMap())
            } catch (e: CancellationException) {
                // Don't catch CancellationException! Re-throw it
                throw e
            } catch (e: Exception) {
                promise.reject("FETCH_ERROR", e.message, e)
            }
        }
    }

    override fun invalidate() {
        scope.cancel() // cancel all pending coroutines
        super.invalidate()
    }
}

// Flows for streaming data (e.g., BLE sensor data to RN)
private val sensorFlow: SharedFlow<SensorData> = bleScanner
    .scan()
    .conflate()   // drop old values if JS can't keep up
    .shareIn(scope, SharingStarted.WhileSubscribed(5000), replay = 1)

JNI & C++ for RN Engineers¶

React Native's JSI uses C++ and JNI extensively. Understanding this unlocks the ability to build performant TurboModules and debug native crashes.

Text
// JSI Host Object: exposes C++ object directly to JavaScript
class NativeCryptoModule : public jsi::HostObject {
public:
  jsi::Value get(jsi::Runtime& rt, const jsi::PropNameID& name) override {
    auto propName = name.utf8(rt);

    if (propName == "hash") {
      return jsi::Function::createFromHostFunction(
        rt, jsi::PropNameID::forUtf8(rt, "hash"), 1,
        [](jsi::Runtime& rt, const jsi::Value&,
           const jsi::Value* args, size_t) -> jsi::Value {
          auto input = args[0].asString(rt).utf8(rt);
          // Direct C++ computation — zero serialization overhead
          return jsi::String::createFromUtf8(rt, sha256(input));
        }
      );
    }
    return jsi::Value::undefined();
  }
};

// Register via JNI
extern "C" JNIEXPORT void JNICALL
Java_com_myapp_NativeCryptoModule_install(JNIEnv* env, jobject, jlong jsRuntime) {
  auto& rt = *reinterpret_cast<jsi::Runtime*>(jsRuntime);
  auto module = std::make_shared<NativeCryptoModule>();
  rt.global().setProperty(rt, "NativeCrypto",
    jsi::Object::createFromHostObject(rt, module));
}
Note
Debugging Native Crashes (tombstones)
When C++/JNI code crashes, Android generates a tombstone in /data/tombstones/. Use ndk-stack to symbolicate: adb logcat | ndk-stack -sym ./app/build/intermediates/cmake/debug/obj/arm64-v8a. For Hermes crashes, use the .map file from your Hermes build to map bytecode offsets.

Staff LevelInterview Q
Explain how a React Native touch event on Android travels from the screen to a JS event handler. What can go wrong at each stage?
Answer: The touch pipeline:
  1. Hardware → InputReader (kernel): Raw touch coords arrive from touchscreen driver via /dev/input
  2. InputDispatcher: Routes to the focused window (your Activity). If main thread is blocked >5s, ANR.
  3. ReactRootView.onTouchEvent(): On Main Thread. Creates a ReactPointerEventsMap, dispatches to JSTouchDispatcher.
  4. Old Arch: Serialized to JSON, queued on bridge, sent to JS thread asynchronously — ~1-2 frame delay.
  5. New Arch (Fabric): Touch is handled synchronously on UI thread via C++ event emitter, JSI call on JS thread — <1 frame latency.
  6. JS: React synthetic event system, bubbling, handler execution.
What goes wrong: Main thread jank blocks input. Long JS frame delays touch response. Fabric fixes this with synchronous event dispatch.
FAANG LevelInterview Q
Your Android app has a 3% crash-free session rate regression after a Gradle upgrade. Walk through your investigation.
Answer:
  1. Check Firebase Crashlytics: group crashes by type. Is it a native crash (signal 11) or Java exception?
  2. Compare R8/ProGuard rules between versions. Gradle upgrades often bump R8 version → more aggressive optimization → missing keep rules.
  3. Check mapping.txt in Firebase to deobfuscate stack traces.
  4. Run ./gradlew :app:generateReleaseR8Usage to see what R8 is removing.
  5. If native crash: check NDK version bump. Different NDK versions have different C++ STL ABIs.
  6. Bisect: revert Gradle version, confirm regression disappears, then add back incrementally.
  7. Write regression test: CI step that verifies crash-free rate on Firebase Test Lab before merge.

ARC Memory Management¶

How ARC Works Internally¶

ARC (Automatic Reference Counting) inserts retain/release calls at compile time — it's not a garbage collector. Objects are freed immediately when their retain count hits zero. This means deterministic deallocation but requires developer awareness of retain cycles.

Text
RETAIN CYCLE (Memory Leak Pattern)

  ViewController ──strong──► ViewModel
       ▲                         │
       └─────────strong───────────┘
                  (cycle!)

  FIX: Use [weak self] or [unowned self]

  class ViewModel {
      var onUpdate: (() -> Void)?  // strong

      func bind(to vc: ViewController) {
          onUpdate = { [weak vc] in   // weak — no cycle
              vc?.updateUI()
          }
      }
  }

REACT NATIVE RETAIN CYCLE PATTERN:
  RCTBridge ──strong──► NativeModule
  NativeModule ──strong──► closure with self
                            self has reference back → LEAK

  FIX: In RCTEventEmitter subclasses, always use
  [weak self] in async callbacks to bridge
Text
// Common RN memory leak: strong capture in async callback
class MyNativeModule: NSObject, RCTBridgeModule {

    // BAD: strong self capture → retain cycle with bridge
    func badFetch(_ resolve: @escaping RCTPromiseResolveBlock) {
        URLSession.shared.dataTask(with: url) { data, _, _ in
            self.process(data)  // strong capture!
            resolve(result)
        }.resume()
    }

    // GOOD: weak self, guard let to safely unwrap
    func goodFetch(_ resolve: @escaping RCTPromiseResolveBlock) {
        URLSession.shared.dataTask(with: url) { [weak self] data, _, _ in
            guard let self else { return } // Swift 5.3+
            self.process(data)
            resolve(result)
        }.resume()
    }
}

RunLoop Internals¶

The iOS RunLoop is conceptually identical to Android's Looper. It's an event-processing loop on a thread. React Native's JS thread has its own RunLoop managed by the bridge. Understanding RunLoop explains why timers, CADisplayLink, and network callbacks all need one.

  RunLoop.run()

      ├─ Check observers (kCFRunLoopBeforeTimers, AfterWaiting, etc.)
      └─ Sleep (waiting for events)

Staff LevelInterview Q
How does React Native integrate with the iOS RunLoop, and how does this affect timer accuracy in Reanimated?
Answer: React Native creates a dedicated JS thread with its own CFRunLoop. The RCTJSThread is started during bridge initialization. JavaScript timers (setTimeout) are implemented as NSTimer callbacks on this RunLoop.

Reanimated v3+ bypasses this entirely: it uses CADisplayLink on the main thread (or a dedicated animation thread) synchronized to the display's vsync signal (60/120Hz). This gives frame-accurate animation timing vs the ±16ms jitter of RunLoop-based timers. The worklets run in a separate JS runtime context with its own RunLoop, isolated from the main JS bundle.

Implication: Don't use setInterval for animations. Always use Reanimated or Animated (which internally also uses the display sync path on iOS).

Fiber Architecture — First Principles¶

Before Fiber (React 15 and earlier), reconciliation was synchronous and recursive. Once React started processing an update, it couldn't stop until the entire tree was processed. On complex trees, this caused dropped frames.

Fiber (React 16+) rewrites reconciliation as an interruptible, incremental work loop. Each component becomes a "fiber" — a unit of work that can be paused, resumed, prioritized, or discarded.

Text
FIBER DATA STRUCTURE

Each Fiber is a plain JS object:
{
  type:        Button          // component type
  key:         null
  stateNode:   DOM/Native node // actual rendered node
  return:      parentFiber     // parent
  child:       firstChildFiber // first child
  sibling:     nextSibling     // next sibling
  alternate:   workInProgress  // double buffering twin
  flags:       Update | Placement | Deletion
  lanes:       0b00010         // priority lane
  memoizedState: { hook linked list }
  pendingProps:  { ... }
  memoizedProps: { ... }
  updateQueue:  { pending updates }
}

FIBER TREE TRAVERSAL (DFS):
  parent
    │
    ├── child1  ← child
    │     └── child1.1 ← child
    └── child2  ← sibling

Order: parent → child1 → child1.1 → child2

FAANG LevelInterview Q
Explain tearing in React concurrent mode. How does React 18 prevent it?
Tearing occurs when multiple components read the same external store, but React renders them at different times during a concurrent render. Component A reads value=1, React pauses, the store updates to 2, Component B reads value=2 — UI is inconsistent (torn).

React 18 solution: useSyncExternalStore. It subscribes to external stores with a snapshot mechanism. React detects if the snapshot changed mid-render and re-renders synchronously. Redux, Zustand, and Jotai all updated to use this in their React 18 adapters.

RN implication: If you're reading from native module state directly (vs through the React reconciler), you can get tearing. Always gate external state reads through useSyncExternalStore or use a proper state management library.

Hermes compiles JavaScript to bytecode at build time (not at runtime). When the app starts, it reads the pre-compiled bytecode directly — no parsing, no JIT warmup. This is the single biggest TTI improvement for RN apps.

Senior LevelInterview Q
What is the difference between microtasks and macrotasks, and how does async/await relate to them?
Microtasks run immediately after the current task completes, before the event loop picks up the next macrotask. Every Promise resolution enqueues a microtask. The microtask queue is fully drained between macrotasks.

async/await is syntactic sugar over Promises. Every await suspends the async function and schedules its continuation as a microtask when the awaited value resolves. So:
Text
async function foo() {
  console.log('A');
  await Promise.resolve();  // schedules continuation as microtask
  console.log('B');
}
foo(); console.log('C');
// Output: A, C, B
In React Native: Heavy async chains (100s of awaits) can starve the render loop. Use InteractionManager or chunk work into macrotasks for UI-critical paths.

class WebSocketManager {
  private ws: WebSocket | null = null;
  private reconnectDelay = 1000;
  private maxDelay = 30000;
  private pingInterval: ReturnType<typeof setInterval>;

  connect() {
    this.ws = new WebSocket(WS_URL);

    this.ws.onopen = () => {
      this.reconnectDelay = 1000; // reset backoff
      this.startHeartbeat();
    };

    this.ws.onclose = () => {
      this.stopHeartbeat();
      // Exponential backoff with jitter
      const jitter = Math.random() * 1000;
      setTimeout(() => this.connect(),
        Math.min(this.reconnectDelay + jitter, this.maxDelay));
      this.reconnectDelay = Math.min(this.reconnectDelay * 2, this.maxDelay);
    };
  }

  private startHeartbeat() {
    this.pingInterval = setInterval(() => {
      if (this.ws?.readyState === WebSocket.OPEN) {
        this.ws.send(JSON.stringify({ type: 'ping' }));
      }
    }, 25000); // below NAT timeout (~30s)
  }
}

CAP Theorem for Mobile¶

C — Consistency
Every read receives the most recent write. Mobile impact: offline writes might be stale when the user goes online. Healthcare apps require strong consistency (patient records).
A — Availability
Every request receives a response (might be stale). Mobile impact: optimistic updates show cached data while real data syncs. Social feeds prefer availability over consistency.
P — Partition Tolerance
System works despite network partitions. Always required for mobile (users lose connectivity). You choose: CP (banking) or AP (social, e-commerce) for mobile scenarios.

Staff LevelInterview Q
Your mobile app has a chat feature. The backend team asks whether to use REST polling, WebSocket, or Server-Sent Events. What do you recommend and why?
Recommendation: WebSocket with SSE fallback.

REST polling: Inefficient — constant requests even when no messages. Battery drain. Works as last resort.

SSE (Server-Sent Events): Server → client only. Good for feeds/notifications but can't send messages back on the same connection. HTTP/1.1 has 6-connection limit (SSE uses one). iOS NSURLSession handles SSE natively but RN needs polyfill.

WebSocket: Full-duplex. Perfect for chat. Challenges: NAT timeouts (heartbeat every 25s), reconnection (exponential backoff), scaling (sticky sessions or pub/sub layer like Redis Pub/Sub). Use socket.io for automatic fallback and reconnection.

Mobile-specific concerns: Background mode — both platforms limit background socket connections. iOS: background task + push to wake. Android: FCM for wakeup, foreground service for active chat. I'd use a dedicated WebSocket connection with offline queue (SQLite) + push notifications for when app is backgrounded.

MVI (Model-View-Intent) for Complex Screens¶

Text
// MVI pattern: unidirectional data flow, testable state machine
// Ideal for complex screens: cart, checkout, multi-step forms

type CartState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'loaded'; items: CartItem[]; total: number }
  | { status: 'error'; message: string };

type CartIntent =
  | { type: 'ADD_ITEM'; item: CartItem }
  | { type: 'REMOVE_ITEM'; id: string }
  | { type: 'CHECKOUT' }
  | { type: 'CLEAR' };

function cartReducer(state: CartState, intent: CartIntent): CartState {
  switch (intent.type) {
    case 'ADD_ITEM':
      if (state.status !== 'loaded') return state;
      const newItems = [...state.items, intent.item];
      return {
        ...state,
        items: newItems,
        total: newItems.reduce((s, i) => s + i.price * i.qty, 0),
      };
    case 'REMOVE_ITEM':
      if (state.status !== 'loaded') return state;
      return {
        ...state,
        items: state.items.filter(i => i.id !== intent.id),
      };
    default:
      return state;
  }
}

// In component
function CartScreen() {
  const [state, dispatch] = useReducer(cartReducer, { status: 'idle' });

  // Pure render from state — highly testable
  return (
     dispatch({ type: 'ADD_ITEM', item })}
      onCheckout={() => dispatch({ type: 'CHECKOUT' })}
    />
  );
}

Crash Debugging Walkthroughs¶

P0 IncidentNative crash on Android 14 — 4% crash rate on launch

Signal: Firebase Crashlytics showed SIGSEGV in libhermes.so at 0x3f2a. Only Android 14 (API 34), arm64.

Step 1: Get symbolicated stack. Download libhermes.so with debug symbols from Hermes GitHub release matching your exact version. Run:

bash
llvm-symbolizer --obj=libhermes.so 0x3f2a
# Output: hermes::vm::JSObject::getNamedSlotValue(...) at Object.cpp:245

Step 2: Narrow to JS code. Enable Hermes source maps: in Metro config, set hermesFlagsRelease: ['-O', '-output-source-map']. Upload source map to Sentry/Firebase.

Step 3: Found — a native module was calling into JSI after the React host was destroyed (app backgrounded → system killed activity → JSI runtime torn down). Module had no nil-check on runtime pointer.

Fix: Added if (!runtime_) return; guard in C++ module. Added JSI invalidation listener. Added CI step that runs address sanitizer on debug builds.

Memory LeakiOS app memory growing 50MB/hr during normal use

Detection: Instruments → Allocations → Memory growth 50MB over 30min session in production. User reports: app killed after ~2 hours.

Investigation:

bash
# Instruments Leaks template → found: RCTImageView not released
# Mark generation: interact with app, mark heap, interact more, compare
# Found: 2,000 RCTImageView instances alive, expected ~20

Root cause: FastImage cache wasn't being cleared. Each navigation push created new images, old ones weren't released because a global cache (a plain JS object) held strong references to image URIs mapped to NativeImage objects.

Fix: Implemented LRU cache with max size 200. Added FastImage.clearMemoryCache() on app background. Switched to Image.getSizeWithHeaders for prefetching instead of loading invisible Image components.

Risks of AI-Generated Code¶

Critical
AI Hallucination Patterns in Mobile Code
1. Invented APIs: AI generates NativeModules.Camera.captureWithOptions() — this API doesn't exist. Always verify against actual docs.
2. Version drift: AI trained on older library versions. React Navigation v5 API vs v7, Reanimated v1 vs v3 patterns are completely different.
3. Platform inconsistencies: AI may generate iOS-only code without Android equivalents, or use deprecated Android APIs.
4. Security omissions: AI-generated auth code often omits PKCE, token rotation, certificate pinning.
5. Test false confidence: AI-generated tests that always pass — mocks so aggressive they test nothing.

  • AI for undifferentiated work: Boilerplate, tests, documentation, migration scripts — high ROI. Business logic, security-critical code — use AI as a collaborator, verify everything.

React Native Core¶

SeniorInterview Q
What is the difference between the Old Architecture and New Architecture in React Native?
Old Architecture: JS ↔ Native communication via async JSON bridge. Layout in JS (YogaJS). Two trees: JS shadow tree + native view tree synchronized asynchronously. Native modules loaded eagerly. All data serialized/deserialized.

New Architecture (JSI + Fabric + TurboModules):
  • JSI: JS directly references C++ objects — no serialization, synchronous calls possible
  • TurboModules: Native modules loaded on-demand via code generation, strongly typed via TypeScript specs
  • Fabric: C++ renderer, synchronous layout, Concurrent Mode support, direct UIKit/View mount
  • Codegen: TypeScript → C++ bindings automatically, type-safe at compile time
Impact: 40-60% startup improvement, sub-5ms native call latency, React 18 Concurrent Mode works correctly.
SeniorInterview Q
How does Metro bundler work? What is its role in the development vs production build?
Metro is RN's JavaScript bundler. It: (1) resolves module imports using its resolver (supports platform-specific files: .ios.js, .android.js), (2) transforms files via Babel (TS→JS, JSX→React.createElement), (3) bundles into a single JS bundle with module map, (4) serves via dev server with HMR in development.

Dev: Serves bundle from Metro dev server. Fast refresh via WebSocket connection. Source maps inline.

Production: Generates index.android.bundle / main.jsbundle. Runs Hermes compiler on the bundle → .hbc bytecode. Source maps generated separately for crash symbolication. Bundle split into RAM bundle format for lazy loading (optional).
StaffInterview Q
How would you design a TurboModule for high-frequency sensor data (1000 readings/second from an accelerometer)?
Key challenge: 1000 events/sec via JSI events would flood the JS thread.

Design:
  1. Native-side batching: Collect readings in a ring buffer on a dedicated native thread. Every 16ms (one frame), flush up to 16 readings as a single batch event to JS.
  2. JSI NativeEventEmitter replacement: Use a direct JSI callback registered once — setDataCallback(callback). Each frame calls the callback with a typed array (Float32Array) directly mapped to C++ memory — zero copy.
  3. Back-pressure: If JS can't keep up (callback not returned before next flush), drop oldest readings, emit a "dropped" count.
  4. Worklet offloading: If processing is numeric (FFT, peak detection), run in a Reanimated worklet on the UI thread, bypassing JS entirely.
  5. API design: startStream(intervalMs, onData, onError) returns a subscription handle. stopStream(handle) cancels the native source.

SeniorInterview Q
What is the difference between compileSdkVersion, targetSdkVersion, and minSdkVersion in Android?
minSdkVersion: Minimum Android version your app supports. Devices below this won't install the app. Set to 24 (Android 7.0) for modern RN apps.

compileSdkVersion: SDK version used to compile your code. Use the latest (34). Gives access to latest APIs and compiler checks. Doesn't affect runtime.

    // Debounce: don't fire on every keystroke
    const timeoutId = setTimeout(async () => {
      // Cancel previous in-flight request
      abortRef.current?.abort();
      abortRef.current = new AbortController();

      setLoading(true);
      setError(null);

      try {
        const data = await searchAPI(query, {
          signal: abortRef.current.signal
        });
        setResults(data);
      } catch (e) {
        if ((e as Error).name !== 'AbortError') {
          setError(e as Error);
        }
        // AbortError is expected on fast typing — ignore
      } finally {
        setLoading(false);
      }
    }, delay);

    return () => {
      clearTimeout(timeoutId);
      abortRef.current?.abort(); // cancel on unmount
    };
  }, [query, delay]);

  return { results, loading, error };
}

// Implement a useInterval hook that pauses on background
function useInterval(callback: () => void, delay: number | null) {
  const savedCallback = useRef(callback);

  // Keep ref up to date (avoids stale closure)
  useEffect(() => {
    savedCallback.current = callback;
  }, [callback]);

  useEffect(() => {
    if (delay === null) return; // null = paused

    const id = setInterval(() => savedCallback.current(), delay);
    return () => clearInterval(id);
  }, [delay]);
}

function LivePrice({ symbol }: { symbol: string }) {
  const appState = useRef(AppState.currentState);
  const [active, setActive] = useState(true);

  useEffect(() => {
    const sub = AppState.addEventListener('change', state => {
      setActive(state === 'active');
      appState.current = state;
    });
    return () => sub.remove();
  }, []);

  useInterval(
    () => fetchPrice(symbol),
    active ? 3000 : null // pause when backgrounded
  );
}

The Staff Engineer Mindset¶

Tip
What Separates Good from Great at the Staff Level
Technical Excellence is table stakes. Every Staff candidate can code. What interviewers at FAANG evaluate beyond coding:

1. Judgment under ambiguity. Given incomplete information, can you drive to a decision? Propose options, state tradeoffs, recommend one.

2. Scope amplification. A Senior engineer solves their task. A Staff engineer asks: "How do I solve this in a way that makes all future similar tasks easier for the entire org?"

3. Written communication. RFCs, postmortems, design docs. Can you influence without being in the room? Your writing is your leverage at scale.

4. Constructive tension. Push back on bad ideas from anyone — PM, engineering manager, senior leadership — with data, not ego. Then execute the final decision fully, even if it's not yours.

5. Make others better. Code review, mentoring, architecture guidance. Your impact is multiplied through the people around you.
